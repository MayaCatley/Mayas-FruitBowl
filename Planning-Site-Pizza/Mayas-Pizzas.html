    Relevant Implications Summary

    Future Proofing

    Future Proofing was essential to ensure that my program would not only function properly now, but in the future too.

    To do this, I made sure that my program was easy to update, I achieved this making sure it was easy to add new functionality and features, I addressed this by adding new menu items and new functions (as my program is a function based build) frequently to ensure no errors would arise. This is why it was so crucial to create the program by pushing each Sprint to Git to ensure that if a function happened to break a section of the program it was easy to go back to the previously working program.

    I also had to ensure that my program would be easy for another developer to work on and improve, I achieved this by making sure that majority of my code was commented with helpful and descriptive sentences briefly explaining the purpose of functions and lines of code, I also included doc typing in my code to document specific segments of my code, doc typing consists of listing the classes, exceptions and functions, and a one-line summary. As well a starting my program with a main function, all of these allow for the code to be easy to understand and simple to see how the code works.


    Functionality

    Functionality was crucial when making sure that my program worked well, and that it had no bugs and wouldn't crash

    To do this I had to ensure that my program worked properly (for example when adding costs), and that it didn't crash, this means that I had to regularly and systematically test my program after each Sprint. I achieved this by getting my friends to test out my program as I went, and then finally getting my Dad to test it (which is the video above). It was so important to test my code regularly and fix errors as they happened, as it would have been so much more difficult to try and fix multiple errors at the same time. Another way in which I dealt with errors, was putting my code through a Python Style Checker, majority of my errors we due to too long lines which was a easy fix, but it was great to know afterwards how few errors my code had. Another step I took to improve my program's functionality was to have separate component tests, this meant that I could trial sections of my code separately, so I could deal with any errors on a smaller scale, rather than trying to put the new code in straight away and all at once.

    Here is an example of some testing I did in regards to phone numbers. This was to test what numbers would be accepted and what numbers wouldn't. It was super helpful to do this test first as I could find errors much easier than searching through all of my code to find errors.



    Usability

    Usability makes sure that the user always knew what was going on.

    To do this I had to ensure my program was easy to use, I did this by making sure the user was always receiving informative user feedback (This was achieved by constantly printing out an updated receipt with the users order, cost total, and choice of delivery/pickup/choose later), I made sure to communicate with the user if they made an error (for example "This doesn't appear to be a valid phone number"), or coding it so my program was able to fix the error on its own; this was done by striping spaces, and converting uppercase and lowercase letters. I also made sure I used a consistent navigation system which can be seen on the main menu and in the receipt.

    I also made sure my program was predictable, I did this by making sure the user could always return to the main menu if they wanted I addressed this by making "enter 0 to exit" always an option. I also made it so the user was always receiving feedback once they completed a task, for example "You have selected to choose Delivery or Pickup later", this allows the user to understand what is currently happening, and not make them guess.
